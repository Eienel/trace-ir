# TRACE - Dataset Documentation

## Demo dataset (ships with the repo)

Generated by `benchmark/make_sample_data.py` into `sample_data/case01/`. It is a
synthetic-but-realistic single-host Windows intrusion, shaped to match the output
of real DFIR tools so the full pipeline runs without the SIFT VM.

| Artifact | Mimics | Malicious content |
|---|---|---|
| `amcache.txt` | AmCache (regipy) | `svch0st.exe` in `\Users\Public`, `update.exe` in `\AppData\Local\Temp` |
| `mft.txt` | $MFT timeline (analyzeMFT) | creation of `svch0st.exe` in `\Public`, timestomping of same |
| `prefetch.txt` | Prefetch | `svch0st.exe` executed 7× |
| `evtx_logons.txt` | Security EVTX (python-evtx) | 5× failed logons + external RDP (type 10) from `203.0.113.45` |

`ground_truth.json` lists the 7 finding IDs a correct analysis must surface. It
exists only to let the benchmark score accuracy objectively; the agent never
reads it.

The scenario models a credential-access → execution → defense-evasion chain
(external brute force → RDP foothold → dropped binary in a world-writable dir →
timestomping to hide it). IPs use the `203.0.113.0/24` / `192.168.0.0/16`
documentation ranges. No real PII or proprietary case data is included.

## Real-evidence run with a real forensic library (python-evtx)

TRACE was also validated against a REAL Windows Security event log using
`python-evtx`, the same EVTX parsing library used on the SIFT Workstation. This
proves the architecture is pluggable: a real forensic tool reads a real binary
artifact, normalizes it, and the unchanged TRACE pipeline (citations, verifier,
loop, logs) analyzes it.

Reproduce:

```bash
# 1. Install the real forensic library
pip install python-evtx --break-system-packages

# 2. Export a real Security log from Windows (elevated PowerShell):
#    wevtutil epl Security "$env:USERPROFILE\Desktop\findevil hac\real_security.evtx"

# 3. Parse the REAL artifact with python-evtx into the case format
python3 -m server.real_tools "/mnt/c/Users/HP/Desktop/findevil hac/real_security.evtx" sample_data/case_real

# 4. Run the SAME TRACE pipeline on the real artifact
python3 -m agent.loop --case sample_data/case_real --max-iterations 3
```

Not a single line of `agent/`, the verifier, or the loop changes between the
synthetic case and the real `.evtx`. Only the parser in front differs.

Real-evidence run log:

- Source artifact: Windows Security.evtx (host: Eienel, 21 MB, exported via `wevtutil epl`)
- Library used: python-evtx 0.8.1
- Logon events parsed: 1,239 (4624 successful: 1,238 / 4625 failed: 1 / type-10 RDP: 0)
- Malicious findings: 0 (no external RDP, no brute-force pattern) - correct for a clean host
- Hallucinations dropped by verifier: 2 (the same uncited "exfiltration" / "attribution"
  claims, caught on real data too)
- Result: TRACE produced ZERO false positives on 1,239 real events and still caught the
  fabricated claims. On a clean machine, the correct answer is "nothing malicious," and
  that is exactly what TRACE reported - it did not invent threats.
